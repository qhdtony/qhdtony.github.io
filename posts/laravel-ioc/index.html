    Laravel Ioc

    August 23, 2025

    一、什么是Ioc容器。

    负责对象的创建和依赖管理，在这个容器里，不需要手动处理类之间的依赖关系。

    IoC 容器的作用
1. 解耦
    * 类不再自己去实例化依赖，而是通过容器自动注入。
    * 方便替换实现（如接口绑定不同的实现类）。
2. 依赖管理
    * 统一在容器中管理依赖关系，避免到处 new。
3. 服务解析
    * 通过容器 make() 方法即可解析并返回对象。

Laravel IoC 容器本质上就是一个 依赖管理器，它维护一个 绑定表（bindings）和一个 实例表（instances），解析时通过：
1. 查看有没有绑定
2. 如果没有绑定，就通过 反射机制 自动实例化对象，并递归解析其依赖
核心文件：
* Illuminate\Container\Container

    二、原理对比理解。

    1、普通写法

    /*
需求：
你想开车出门。
你得自己造一台车（发动机、轮胎、车门都要自己组装）。
想换车？那你又得重新造一台
即需要动业务代码
*/
class Driver {
    public function __construct() {
        $this->car = new BMW();//每次都要改类
    }


    public function drive() {
        $this->car->run();
    }
}

(new Driver())->drive();

Driver 固定绑定 了 BMW，想换成 Tesla 必须改源码。

这样写也是可以的，为了创建灵活的环境，可以直接在Driver::__construct()的构造函数里添加字符参数。
类似于:
class Driver {
    public function __construct($carname) {
        $this->car = new BMW($carname);//每次都要New
    }
    public function drive() {
        $this->car->run();
    }
}
(new Driver("civic"))->drive();

但是框架不会这样简单考虑，因为一个字符串还要去维护（例如放到数组里），没有显示表达出来。
这里说的显示表达类似于laravel里面的：
$this->app->bind(Car::class, Tesla::class);

    2、控制反转写法

    <?php

//需求：我要一辆车，啥牌子无所谓，能跑就行。
//你只负责开车，不关心车是怎么造出来的。

//切换实现的时候，只改 绑定关系，不用动业务代码。
Interface Car {
    Public function run();
}


Class CT5 implements Car {
    Public function run() {
        echo "CT5 run";
    }
}


Class Accord implements Car {
    Public function run() {
        echo "Honda run";
    }
}


Class Driver {
    Protected $car;
    Public function __construct(Car $car) {
        $this->car = $car;
    }
    Public function drive() {
        $this->car->run();
    }
}


(new Driver(new CT5()))->drive();//这样写会很复杂。

替换成类似 $this->app->bind(Car::class,CT5::class); laravel的写法 就比较清晰。

    三、Laravel 中 怎么用

    1、绑定和解析

    use App\Services\UserService;
use App\Services\Contracts\UserServiceInterface;


// 在服务提供者中绑定（AppServiceProvider 或自定义）
$this->app->bind(UserServiceInterface::class, UserService::class);


// 在控制器或其他地方解析
$userService = app(UserServiceInterface::class);
// 或者
$userService = resolve(UserServiceInterface::class);

/*
bind()：每次解析返回一个新实例
singleton()：全局只会有一个实例（单例）
instance()：绑定一个现成的对象
*/

    2、自动依赖注入

    Laravel 容器可以自动解析构造函数依赖：
class OrderController extends Controller
{
    protected $userService;


    public function __construct(UserServiceInterface $userService)
    {
        $this->userService = $userService;
    }


    public function index()
    {
        return $this->userService->all();
    }
}

OrderController 构造函数依赖了 UserServiceInterface
Laravel 会自动从容器中解析绑定的实现 UserService 并注入

    3、闭包绑定

    $this->app->bind('foo', function ($app) {
    return new FooBar($app['config']['foo']);
});
$foo = app('foo');
这种方式适合需要运行时动态创建对象的场景。

    4、上下文绑定

    在不同场景下为同一个接口提供不同实现：
use App\Services\SmsSender;
use App\Services\EmailSender;
use App\Contracts\Notification;


$this->app->when(OrderController::class)
          ->needs(Notification::class)
          ->give(SmsSender::class);


$this->app->when(UserController::class)
          ->needs(Notification::class)
          ->give(EmailSender::class);

OrderController 注入的是 SmsSender
UserController 注入的是 EmailSender

    5、标签

    多个类打标签，一次性解析。

$this->app->bind('Mailer', SmtpMailer::class);
$this->app->bind('Notifier', SlackNotifier::class);

$this->app->tag(['Mailer', 'Notifier'], 'messaging');

$services = $this->app->tagged('messaging');

    四、在Laravel哪里用。

    1、对外依赖接口，不依赖具体实现（面向接口编程）

    2、业务逻辑解耦，方便单元测试

    3、在 ServiceProvider 中集中绑定依赖

    4、尽量用依赖注入而不是 Facade，可测试性更强

    五、适用IoC的场景

    1. 依赖可能会替换/扩展
    * 支付：支付宝 / 微信 / PayPal
    * 通知：短信 / 邮件 / Slack
    * 存储：本地 / AWS S3 / 阿里云 OSS
2. 需要写单元测试
    * 测试时，可以用 IoC 注入一个假的实现（Mock），不用依赖真实服务。
    * 例如：用 FakePaymentService 替换真实的支付接口。
3. 多人协作/大项目
    * 不同团队负责不同模块，通过接口约定解耦，避免相互修改代码。
4. 依赖复杂，嵌套层级多
    * A 依赖 B，B 依赖 C …
    * 用 IoC 容器时，Laravel 会自动通过反射帮你解析依赖，少写很多 new。

    六、为什么用控制反转

    这是 以空间换时间 或 以复杂换灵活 的典型例子。

场景小、变化少 → 直接 new，没问题，成本低。
场景大、依赖多、可能替换实现 → IoC 就能显著降低维护成本。

举个例子：
* 如果你写的是一个 小工具脚本，就 1 个类，new 绝对划算。
* 如果你写的是 电商系统：
    * 支付接口可能有 支付宝 / 微信 / PayPal
    * 发货接口可能有 顺丰 / 圆通 / 京东物流
    * 通知可能有 短信 / 邮件 / Slack

👉 如果不用 IoC，你的代码里到处是 if-else 或者 switch，改一次需求要全局搜代码；
👉 用 IoC，只需要 改绑定关系，甚至可以根据配置文件或环境变量动态切换实现。

    七、IoC 和 DI 的关系和区别

    IoC是一种思想，核心是“谁来负责依赖的创建”。 为什么

    DI 是一种具体实现手段，把依赖从外部传进来。 怎么做

    八、DI 常用方法

    1、构造函数注入（最常用）

    class Driver {
    protected $car;


    // Car 由外部传入，而不是自己 new
    public function __construct(Car $car) {
        $this->car = $car;
    }
}

    2、Setter注入

    class Driver {
    protected $car;
    public function setCar(Car $car) {
        $this->car = $car;
    }
}

    3、方法注入。

    class Driver {
    public function drive(Car $car) {
        $car->run();
    }
}
